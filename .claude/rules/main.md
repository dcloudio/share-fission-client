# AI 工作规则

## 权限
完全权限执行任何命令并自动修复错误。因为代码都通过 git 存好了，即使你误操作我可以通过 Git 回退，但是你禁止操作远端仓库。

## 工作模式

### 交互模式(默认)
技术选型、多方案、重要决策时使用 `AskUserQuestion` 询问用户。

### 无人值守模式
触发词:"进入无人值守模式"
- 直接选择最佳实践方案
- 禁止使用 `AskUserQuestion`
- 自动修复错误和优化

### 谋定而后动
触发词:"谋定而后动"、“先讨论”
含义：指的是不要立即开展行动，先一起分析讨论规划后再确认行动方案再开展工作

## 注意声明
- 辩证客观，不要我提什么直接执行，或者直接说：是的、你是对的...需要加以思考。除非我告诉你：你错了、必须按 xxx 执行、请直接xxx
- 技术强迫症，使用技术前先明确目标不要过度设计，以项目目标为准实施。
- 禁止使用 chrome mcp 等会导致大量 token 浪费的 mcp，如必须使用务必提醒我

## AI 行为规范（必须遵守）

### 规范1：禁止占位实现
❌ 不允许：函数只返回成功但不真正工作
✅ 要求：必须真正调用底层实现/库并产生正确结果。但开发期间为临时跳过部分技术障碍可以写上 TODO 声明，并创建技术债检查单，待整体功能跑通后处理（推荐尝试以“刺探”的方式解决，并总结经验教训到项目文件）

### 规范2：必须实际测试验证
强制流程：
1. 修改代码并保存
2. 运行平台：`npm run dev:web` / `npm run dev:app-android` / `npm run dev:app-ios` / `npm run dev:app-harmony` 等（具体命令见 package.json scripts）
3. 查看日志：`npm run logcat:web` / `npm run logcat:app-android` / `npm run logcat:app-ios` / `npm run logcat:app-harmony` 等（具体命令见 package.json scripts）
4. 确认修复生效
5. 日志确认功能正常后才能说"修复完成"

原因：编译通过≠运行正常，很多问题只在运行时暴露

### 规范3：避免过度工程
❌ 不要：添加未要求的功能、给未修改的代码添加注释、为不会发生的场景添加错误处理
✅ 只做：用户明确要求的功能，保持简单


## 生命周期

### 1. 启动阶段
启动开发服务（具体命令见项目规则文件，通常为 `npm run dev:web` 或类似）

**日志获取方式:**
- **编译器日志**: 从开发服务后台进程读取（编译、HMR、语法错误等）
- **运行日志**: 使用 `npm run logcat:web` / `npm run logcat:app-android` / `npm run logcat:app-ios` 等获取控制台输出（运行时错误、console.log 等，具体命令见 package.json scripts）

### 2. 开发阶段
**功能开发:**
1. 理解需求 → 2. 探索代码 → 3. 实现 → 4. 查看编译器日志验证

**调试错误:**
1. 查看编译器日志 - 检查语法错误、模块解析失败等
2. 使用 `npm run logcat:web` / `npm run logcat:app-android` / `npm run logcat:app-ios` 等查看运行日志 - 检查运行时错误（具体命令见 package.json scripts）
3. 定位代码 → 修复 → 验证

**错误排查流程:**
遇到错误 → 查看编译器日志 → 查看运行日志 → 定位代码 → 修复 → 重新运行验证

**代码修改不生效检查清单:**
1. 代码是否保存
2. 编译是否成功
3. 是否重新启动开发服务器
4. 缓存是否清理（npm run 命令通常自动清理）

### 3. 验证阶段

**完成标准（DoD）- 核心原则：实际运行成功才算完成，编译通过 ≠ 完成**

必须满足：
1. 代码编写完成
2. 编译通过
3. 实际运行（启动应用 + 查看日志）
4. 功能验证（日志确认功能正常）
5. 无运行错误
6. **记录经验教训**（如有）

**关键：** 日志显示功能被调用且返回正确结果才算完成

**验证完成后的强制检查清单**：
- [ ] 功能已验证通过
- [ ] **是否有经验教训需要记录？**（技术难点、踩坑经验、调试技巧、最佳实践等）
- [ ] 如有经验教训，已添加到 `.claude/rules/lessons-learned.md`

**编译验证:**
使用 Bash 工具重新启动开发服务，触发重新编译，然后检查编译器日志，确认编译成功、无错误。

**功能验证（重要）:**
完成核心功能修改后，**必须自动运行测试验证**（具体测试流程见项目规则文件）

**任务完成检查清单:**
- [ ] 编译通过
- [ ] 实际运行成功
- [ ] 日志确认功能正常
- [ ] 无运行时错误
- [ ] 功能按预期工作

### 4. 关闭阶段
停止开发服务（具体命令见项目规则文件，通常为 `lsof -ti :<port> | xargs kill -9`）

## 经验教训记录原则（极其重要）

### 记录时机：达到 DoD 后立即检查并记录

**在达到 DoD 之前，禁止记录经验教训！**

原因：
1. **未完成的任务 = 未经验证的假设**
   - 编译通过但未运行，所有"经验"都可能是错的
   - 基于错误假设的"教训"会误导未来的开发

2. **只有运行成功的经验才可靠**
   - 功能真正运行起来，才能确认哪些做法是对的
   - 日志显示正确结果，才能验证实现方案的正确性

3. **过早记录的危害**
   - 记录了大量未经核实的"经验教训"
   - 下次遇到类似问题时，被错误的"经验"误导
   - 浪费时间在错误的方向上

**正确做法（标准流程）：**
1. ✅ 完成任务并达到 DoD（运行成功）
2. ✅ **立即主动检查**：是否有经验教训需要记录？
   - 遇到了技术难点或踩坑？
   - 发现了最佳实践或优化技巧？
   - 调试过程中有重要发现？
   - 解决了非显而易见的问题？
3. ✅ 如有经验教训，**立即记录**到 `.claude/rules/lessons-learned.md`
4. ✅ 记录时注明验证方式（如：日志显示、功能测试通过）

**禁止做法：**
- ❌ 不要在编译通过但未运行时记录"经验"
- ❌ 不要在遇到问题但未解决时记录"解决方案"
- ❌ **不要等用户提醒才记录经验教训**

### 什么值得记录为经验教训？

- ✅ 非显而易见的技术问题及解决方案
- ✅ 容易踩的坑及规避方法
- ✅ 性能优化技巧
- ✅ 调试技巧和工具使用方法
- ✅ 框架/库的特殊用法或注意事项
- ✅ 代码风格和最佳实践
- ❌ 显而易见的常识性内容
- ❌ 一次性的特定业务逻辑

## 关键规则
1. 修改代码前必须先读取
2. 只做必要修改,避免过度工程化
3. 使用 `npm run logcat:web` / `npm run logcat:app-android` / `npm run logcat:app-ios` 等查看运行日志（具体命令见 package.json scripts）
4. **完成核心功能修改后，必须主动运行自动化测试验证**（不要等用户提醒，具体测试流程见项目规则文件）
5. 执行测试前必须先查看项目测试文档获取最新的测试指南和正确的脚本名称
6. 我跟你提过的除业务需求以外的内容，包括并不仅限于：工作方式方法、注意事项、沟通协作流程、风险规避要点、执行标准细则等，有用的部分要提炼为经验教训、纳入工作指引，或是更新到项目文件中，避免再犯同样的错误。
